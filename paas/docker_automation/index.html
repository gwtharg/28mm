Docker Automation

Logstash Example

This document describes an example of the configuration and deployment of a Docker image in a RHEL environment, from a registry URL to the PaaS. It also goes over the CI/CD/Platform tools used in the process. Logstash is part of Centralised Logging on the platform, and is used to collect and manipulate system and application logs sent by fluentd log collectors

It resides on the PaaS and ships its logs to an external set of Elasticsearch VMs, currently in non-prod. This gives us a central way to manage and view large amounts of information in a quick fashion, using the Kibana front end. If you are interested have a read about the ELK stack .

The files will match the layout of this working project . There will be a note of what to change, should you want to build your own template.

Requirements:

  • Access to the non-prod OpenShift environment & relevant platform components (if you wish to replicate)
  • Some understanding of: Elasticsearch (ELK stack), Red Hat Linux, Gradle, Jenkins, Docker, Python, OpenShift

Github layout:

We will use a standard for folder structure and Gradle commands to publish a copy of the image, assuming our project is called ‘myproject’

myproject/caboodle/
myproject/src/
myproject/src/docker/
myproject/src/templates/openshift/

Gradle files:

To save some space & time here, I’ll refer to a working example for these files and mention what to change:

  • This file is our ‘root’ build, it refers to build and publish tasks contained in other build.gradle files, base repos, and any GoCD version information, which we’ll ignore for now.

myproject/build.gradle

  • This file describes our dependencies, registry information and publishing to Caboodle/Nexus. Under dependencies we will just set a dependency on openshift-project to demonstrate. Our Logstash instance will only build once an instance of openshift-project becomes available.

/Caboodle/build.gradle

  • This file describes the docker build itself

/src/docker/Dockerfile

Other config files:

  • /openshift/template.json - How the OpenShift project should be configured, there are a few extra services configured here that can be ignored, but if building your own, replace references of ‘logstash’ with your project name. It also contains Chieftain env variables which are explained later.
  • myproject/settings.gradle - Defines our included submodules
  • myproject/gradle.properties - Options to configure process used for the build. For a Docker image, change it to something unique (so the Caboodle Logstash isn’t overwritten), such as:
...
artifactName=myproject_logstash
description=a test config of logstash
...

More info on the use of these two files .

Docker config:

  • Dockerfile the config for building the image
  • logstash.conf for the base Logstash config required for startup, containing fields replaced by the Python script
  • logstash_entrypoint.py is the script to pull Logstash Chieftain config & replace config values
  • pip.conf has the Caboodle URL for installing Python libraries
  • requirements.txt for the actual Python modules

A Docker image is what Logstash will be built and run as our container, to be eventually deployed on OpenShift. The Gradle components are needed to actually build and publish artifacts (including our image), and now we need to configure the image:

A rhel7 image is used to give us the ability to install Logstash and configure Python libraries required for configuration management. /src/docker/Dockerfile/ also refers to requirements.txt in the parent directory, pointing torequests and pyyaml for REST calls to Chieftain and creation of a .yml file for our Logstash instance respectively.

Logstash is installed, followed by Python & the module installer pip, Java (Logstash needs 1.8) and the libraries defined in requirements.txt.

Install details:

Installation of Logstash and the necessary Python components

RUN yum install -y logstash python2 python2-pip.noarch java \ && pip install -r /tmp/requirements.txt

The entrypoint is the default command to run when the Docker image starts, here we use the Python script itself. It is run to instrument the configuration of Logstash using our configuration data before the Logstash binary file is run, to ensure we have the most recent snapshot.

ENTRYPOINT ["python","/logstash_entrypoint.py"]

Logstash requires a configuration file logstash.conf to start up, and it needs to point to a bunch of external Elastic non-prod VMs with some authentication, filtering and index config. The script will replace tagged fields in the local file with its Chieftain config, prior to copying it into the default Logstash config directory, and finally starting the Logstash executable that points to this config file.

Chieftain configuration:

If we want to build and test the image locally (who doesn’t), we need the Chieftain instance and token id, which is automatically generated as environment variables when we build using Composer for an openshift-project. CHIEFTAIN_INSTANCEID, CHIEFTAIN_TOKENID have been provided in the script in debug mode so can be replaced for testing, otherwise they’re fetched as environment variables.

Refer to this working project to create a unique service component in Chieftain; create an instance and then grab the id. Refer ‘Logstash’ in Chieftain as an example.

Building:

To test locally, build the logstash image:

docker build -t myproject .

To run (interactively):

docker run -it myproject

Testing:

If part of the build fails, it’s useful to run the container and investigate at that point, since we might back out too early (with a failed container) in later steps, e.g.

Step 9 : COPY logstash_entrypoint.py /
 ---> Using cache
 ---> 3e14adcc988f

docker run -it 3e14adcc988f

Otherwise, executing a shell on the built image is good to see if everything’s in place, grabbing the container ID with the first command

docker ps

docker exec -it <container id> /bin/bash

Jenkins config:

Again the document here is great for describing the CI build job that will upload the Logstash artifacts (including the Docker image) to Nexus. I recommend using it to create the job. This is the Jenkins build used for the project as an example.

Composer:

Next we need to Compose! This assumes we have:

  • A complete & configured git layout
  • A successful Jenkins build
  • A Chieftain Logstash service component

Go to Composer and type in the name of your service component, and search the group ‘docker’ which was defined already in the build config. We can ignore tags for now (without a specific tag it will default to non-prod)

  • Add a scale value of 1
  • Add an OpenShift project name ‘logstash-test’

The example json when ‘Compose’ is selected should look like the below (versions may change). We are creating a new instance of openshift-project; if we want to deploy again in the same environment, we can reference an ‘existing’ openshift_project - with tags ‘non-prod’ in Chieftain.

After a while Composer will give us a correlation ID, in the top right of the box, pasting it into Chieftain will provide us with details of the openshift-project and logstash instances now tied to it.

{
 "maven_name": "com.bank.docker:logstash:0.0.45",
 "scale": [
   {
     "name": "logstash",
     "value": 1
   }
 ],
 "new_openshift_project_name": "logstash-test"
}

GoCD:

In GoCD we’ll hopefully see a full set of green, where the trigger is the call by composer, the OpenShift project is created, and finally Logstash is deployed.

And in non-prod OpenShift, the project which has successfully deployed!
